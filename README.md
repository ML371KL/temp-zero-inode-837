# Сейсмостанция «Разлом-26» — сайт с автообновлением

Панель мониторинга режима рынка США. Страница мгновенно читает готовый
`snapshot.json`, который каждые 20 минут собирает GitHub Actions —
поэтому при заходе с любого устройства все данные уже прогружены,
а ключи FRED/Finnhub живут только в секретах репозитория.

## Развёртывание за 10 минут (нужен только аккаунт GitHub)

1. **Создайте публичный репозиторий** (github.com → New repository →
   Public → Create). Публичный — чтобы Actions были бесплатны без лимитов.
2. **Загрузите содержимое этой папки** как есть (Add file → Upload files →
   перетащите всё, включая скрытую папку `.github`; проще всего — залить
   распакованный архив целиком) → Commit.
3. **Секреты**: Settings → Secrets and variables → Actions → New repository
   secret. Один обязательный и два рекомендуемых:
   - `FRED_KEY` (обязательный) — бесплатный ключ с fred.stlouisfed.org
     (My Account → API Keys)
   - `FINNHUB_KEY` (рекомендуется) — бесплатный ключ с finnhub.io (для
     новостных детекторов; без него панель работает: лента и новостная нога
     сканера будут пустыми, но дивидендная data-нога BDC-детектора
     собирается без ключа)
   - `OPENROUTER_KEY` (рекомендуется, v4.10) — ключ openrouter.ai: серверный
     LLM-судья при каждой сборке классифицирует новостные триггеры
     (факт/мнение), и страница получает проверенные события БЕЗ ввода
     ИИ-ключа во вкладке. Судятся только новые кандидаты (обычно 0–3 за
     прогон); модель задаётся переменной репозитория `OPENROUTER_MODEL`
     (Settings → Secrets and variables → Actions → вкладка Variables), а при
     её отсутствии берётся запасное значение из
     `.github/workflows/snapshot.yml`
     (в комплекте — бесплатная; учтите, free-модели OpenRouter могут
     рейт-лимититься — сбой не теряет вердиктов, кандидаты досуживаются
     следующей сборкой; платная haiku стоит доли цента в день). Поставьте
     лимит на ключ для страховки. Без секрета
     детекторы работают консервативнее: мнения отсекаются правилами, а
     «неподтверждённые» события требуют совпадения по двум тикерам.
4. **Первый снимок**: вкладка Actions → слева «Обновление снимка данных» →
   Run workflow. Через ~1 минуту в `docs/` появится `snapshot.json`.
5. **Включите сайт**: Settings → Pages → Source: «Deploy from a branch» →
   Branch: `main`, папка `/docs` → Save.
6. Через пару минут сайт доступен по адресу
   `https://ВАШ_ЛОГИН.github.io/ИМЯ_РЕПОЗИТОРИЯ/` — добавьте в закладки
   на телефоне и компьютере.

## Что важно знать

- **Такт**: базово каждые 20 минут; в торговые часы США (13–21 UTC, будни) —
  каждые 10 (вторая cron-строка). Внешнему будильнику достаточно базового такта (раз в 20 минут): быстрая полоса торговых часов уже встроена в родное расписание, дублировать её снаружи не нужно.
- **Интрадей (WTI, USD/JPY)** добывается сервером через Yahoo Finance
  chart-API (двойной хост query1/query2, ≈15-мин задержка) с хвостовым
  резервом Stooq; интрадей-VIX нулевым эшелоном идёт с официального
  delayed-CDN CBOE (Yahoo — резерв); крипто-резерв при недоступности CoinGecko/Binance —
  Yahoo (BTC-USD; золото — PAXG-USD, резерв GC=F). Страница читает всё из снимка мгновенно.
- **Свежесть**: в шапке страницы написано «снимок сервера · ЧЧ:ММ · N мин
  назад». Если снимок старше полутора часов, страница подсветит это —
  загляните во вкладку Actions (обычно причина — упавший внешний источник,
  следующий запуск чинит сам).
- **Ключ ИИ** (для кнопки «Спросить ИИ») вводится прямо на странице на каждом
  устройстве и НИКУДА не публикуется: запрос уходит с вашего устройства
  напрямую провайдеру. Не вписывайте ИИ-ключи в файлы публичного репозитория.
- **Периодичность**: две строки cron в
  `.github/workflows/snapshot.yml` (минуты выбирайте «неровные», как в
  комплекте — 6,26,46: ровные :00/:20/:40 у GitHub страдают сильнее всего).
- **Приватность**: снимок содержит только рыночные данные, без ключей.
- Локальная версия панели (со своими ключами, без сервера) — отдельный файл
  `market-fault-monitor.html`, они не конфликтуют.

## Безопасность: карта ключей и доступа

**Где живёт каждый ключ.** `FRED_KEY`, `FINNHUB_KEY` и `OPENROUTER_KEY`
(серверный LLM-судья новостей) — только в секретах
репозитория (Settings → Secrets): GitHub маскирует их в логах, файлы репозитория
и `snapshot.json` их не содержат (снимок — только рыночные данные и текстовые
вердикты «факт/мнение» без ключей), а форки
чужих людей к секретам доступа не имеют. На `OPENROUTER_KEY` поставьте лимит
расходов (openrouter.ai → Keys → Credit limit). Ключи ИИ (OpenRouter / Muse Spark) вводятся в поле на
странице; запрос уходит с устройства напрямую провайдеру по HTTPS и нигде не
публикуется. По умолчанию ключ живёт только в памяти вкладки; галка «запомнить
на устройстве» (opt-in) сохраняет его в localStorage браузера. Важно для
github.io: localStorage общий на весь поддомен `логин.github.io` — любой ваш
БУДУЩИЙ Pages-проект на том же логине сможет прочитать сохранённый ключ.
Лечение: не включать галку на этом хостинге, либо отдельный логин под панель,
либо свой домен; плюс лимит расходов на ключ в OpenRouter — обязательная
страховка. **Никогда не вписывайте ИИ-ключи в файлы этого публичного
репозитория** — в опубликованном `docs/index.html` поля ключей намеренно пусты.
Дополнительная страховка: в OpenRouter на ключ можно поставить лимит расходов
(Keys → Credit limit), и даже утёкший ключ не потратит больше заданной суммы;
ротация любого ключа — две минуты.

**Нужна ли защита самой страницы.** Нет: это статичная страница с публичными
рыночными данными, без секретов и без возможности что-либо изменить; посторонний
посетитель не получает ничего, кроме того, что и так публикуют FRED и ФРС.
Если хочется именно приватности (чтобы страницу не видели вовсе): GitHub Pages
пароли не поддерживает — бесплатный путь это Cloudflare Pages + Cloudflare
Access (вход по e-mail-коду, бесплатного тарифа хватает для личного
использования; подключается к этому же репозиторию). Учтите также, что
публичный репозиторий виден в вашем профиле GitHub — при желании дайте ему
нейтральное имя. И не выдавайте права на запись в репозиторий посторонним:
доступ на запись = доступ к секретам через правку workflow.

**Домены.** Покупать ничего не нужно: адрес `https://логин.github.io/имя/`
полноценен, HTTPS включается автоматически. Свой домен — чисто косметическая
опция (порядка $10–15/год у любого регистратора): Settings → Pages → Custom
domain, CNAME-запись у регистратора — сертификат GitHub выпустит сам; бонус —
адрес не раскрывает ваш логин.

**Локальный файл** `market-fault-monitor.html` (не из этого репозитория)
содержит ваши FRED/Finnhub-ключи в коде — не публикуйте его; при случайной
утечке просто перевыпустите оба бесплатных ключа.


## Если расписание «молчит» (нормальное поведение планировщика GitHub)

Ручной запуск сработал, а автоматических всё нет? Это известная особенность,
а не поломка вашей настройки: GitHub распознаёт **новое** cron-расписание
с задержкой от 15 минут до часа с лишним, и первый автозапуск случится лишь
в ближайший тик после распознавания. Дальше тоже без иллюзий: cron у GitHub —
«имеет право запуститься раз в ~20 минут», а не секундомер; дрожание ±5–20 минут
под нагрузкой — норма, ровные минуты (:00/:20/:40) страдают сильнее всего,
поэтому в этом комплекте расписание стоит на :06/:26/:46.

Как «пнуть» планировщик: любой коммит в ветку main пересинхронизирует
расписания (совет сотрудника GitHub) — достаточно отредактировать любую строку
README через веб-редактор и сохранить. Дальше система самоподдерживающаяся:
каждый успешный запуск сам коммитит снимок в main и тем самым держит
планировщик в тонусе.

Как выглядит здоровое состояние: во вкладке Actions запуски появляются
каждые ~20–35 минут (событие schedule), в коммитах — «снимок … UTC» от бота,
а возраст в шапке страницы гуляет от 0 до ~35 минут. Бить тревогу стоит,
только если за 2–3 часа не появилось ни одного автозапуска: тогда откройте
страницу воркфлоу — не показывает ли GitHub плашку об отключении — и нажмите
Enable/Run workflow.

Предупреждение «Node.js 20 is deprecated» в аннотациях запуска безвредно
(экшены принудительно исполняются на Node 24); в текущей версии комплекта
воркфлоу обновлён на Node 24, и предупреждение исчезает.


## Внешний будильник — гарантированное автообновление (5 минут)

Известный класс проблемы GitHub: у отдельного репозитория планировщик может
молчать сутками, хотя ручные запуски работают и вся конфигурация верна
(подтверждённые кейсы — 12+ часов тишины; официального лекарства нет, совет
инженерных гайдов — внешний триггер). Решение: бесплатный внешний cron
дёргает запуск через API. Дубли не страшны: в воркфлоу стоит
`concurrency: snapshot` — если родной планировщик проснётся, наложения
погасятся сами.

**Шаг 1. Токен (ключ-кнопка, умеет только запускать Actions этого репо).**
GitHub → Settings (профиля) → Developer settings → Personal access tokens →
Fine-grained tokens → Generate new token. Repository access: *Only select
repositories* → выберите этот репозиторий. Permissions → Repository
permissions → **Actions: Read and write** (больше ничего). Срок — на ваш вкус
(можно No expiration). Скопируйте токен (`github_pat_…`).

**Шаг 2. Будильник.** Зарегистрируйтесь на cron-job.org (бесплатно) →
Create cronjob:
- URL: `https://api.github.com/repos/ВАШ_ЛОГИН/ИМЯ_РЕПО/actions/workflows/snapshot.yml/dispatches`
- Schedule: каждые 20 минут (в самом воркфлоу — минуты 6,26,46)
- Advanced → Request method: **POST**
- Headers:
  `Authorization: Bearer github_pat_ВАШ_ТОКЕН`
  `Accept: application/vnd.github+json`
  `X-GitHub-Api-Version: 2022-11-28`
- Request body: `{"ref":"main"}`

Нажмите Test run — успех это код **204** (пустой ответ), и через ~минуту в
Actions появится новый запуск «Обновление снимка данных». Готово: обновления
идут независимо от настроения планировщика GitHub.

Безопасность: токен умеет только запускать Actions одного репозитория —
ни читать код, ни менять файлы, ни видеть секреты; при малейшем сомнении
удаляется в Developer settings за два клика. Проверка с компьютера (по
желанию): `curl -i -X POST -H "Authorization: Bearer ТОКЕН" -H "Accept: application/vnd.github+json" https://api.github.com/repos/ЛОГИН/РЕПО/actions/workflows/snapshot.yml/dispatches -d "{\"ref\":\"main\"}"`.

Уточнение про предупреждение «Node.js 20 is deprecated» в аннотациях: оно
относится к внутренностям стандартных экшенов checkout/setup-node (GitHub
принудительно исполняет их на Node 24 — видно в самом тексте) и на работу
не влияет; исчезнет само, когда GitHub обновит мажорные версии экшенов.
